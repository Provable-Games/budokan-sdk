# Spec: group bracket tournaments into one card (Budokan client)

**Audience:** an engineer/agent implementing this in the **Budokan client**
(budokan.gg) — and the small **budokan-sdk** producer change it depends on.

## Problem

A "bracket" is an off-chain 1v1 single-elimination tournament built **on top of
many Budokan tournaments — one tournament per match**. A 4-player bracket =
3 tournaments (2 semis + 1 final); 8 players = 7; 16 = 15.

Today the client renders each match tournament as its own card, so a single
bracket looks like 3/7/15 unrelated tournaments. We want the client to
**recognise tournaments that belong to the same bracket and stack them into one
card** — collapsed: a single bracket summary; expanded: the bracket tree.

## Background: how brackets are created

Source of truth: `budokan-sdk` `src/brackets/index.ts`. Key facts the client
can rely on:

- **Single-elimination, power of two.** Bracket size `N ∈ {2,4,8,16,32}`.
  Matches = `N − 1`. Rounds = `log2(N)`.
- **Round `r` (1-indexed) has `N / 2^r` matches.** Round 1 = `N/2` matches,
  … final = round `log2(N)` with 1 match.
- **Match `m` in round `r` is fed by matches `2m` and `2m+1` of round `r−1`**
  (0-indexed `m`). The two feeders' winners are the two entrants.
- **Deployed bottom-up** (round 1 first), because round `≥2` matches are
  **gated**: their on-chain `entry_requirement` is a `tournament_validator`
  extension that references the **feeder tournament ids** — so a player can only
  enter if they won a feeder. (This means the feeder tournament ids are known at
  the time each later match is created.)
- Each match is an ordinary **2-player Budokan tournament** with:
  - `metadata.name` (felt252, ≤31 ASCII): `"<gamePrefix> R<round>-<match+1>"`
    e.g. `"NumberGuess R1-2"`.
  - `metadata.description` (ByteArray, arbitrary length):
    `"Bracket <bracketId> - round <round>, match <match+1>"`.
  - a spare `metadata_value` felt252 (currently `0`).
- **Prizes are spread across the matches by placement tier** (not all on one
  tournament): champion → final position 1; runner-up → final position 2;
  semifinalists → each semifinal position 2; quarterfinalists → each QF
  position 2; etc. The bracket's total pool is therefore the **sum of prizes
  across all its matches**.

`bracketId` is a stable per-bracket string (e.g. `bmqv0c1qm`) generated by the
bot; it is identical across every match of one bracket and unique per bracket.

## Linking data — how the client groups matches

### Option A (works today, no SDK change): parse the description

Every match description contains `Bracket <bracketId> - round <r>, match <m>`.
The client can:
1. Regex `^Bracket (\S+) - round (\d+), match (\d+)$` → `{ bracketId, round, match }`.
2. Group tournaments by `bracketId`.
3. Infer size from the round-1 match count (`size = 2 × (#round-1 matches)`), or
   from the max round (`size = 2^maxRound`).

Limitation: prose parsing, and the feeder→match edges must be inferred from the
`2m / 2m+1` rule rather than read directly. Good enough for an MVP.

### Option B (recommended): a structured descriptor (small SDK change)

Have the SDK append a single machine-readable line to `description`, so the
client never regexes prose and gets the tree edges explicitly. Format (pipe-
delimited, versioned):

```
bkt|1|<bracketId>|<size>|<round>|<matchIndex>|<rounds>|<isFinal>|<feederTids>
```

| field        | meaning                                                        |
|--------------|---------------------------------------------------------------|
| `bkt`        | literal marker                                                |
| `1`          | schema version                                                |
| `bracketId`  | grouping key (string, no `|`)                                 |
| `size`       | bracket size N (power of two)                                 |
| `round`      | 1-indexed round                                               |
| `matchIndex` | 0-indexed match within the round                              |
| `rounds`     | total rounds = log2(N)                                        |
| `isFinal`    | `1` for the final match, else `0`                             |
| `feederTids` | comma-separated feeder **tournament ids** (empty for round 1) |

Examples (4-player bracket, tournament ids 6,7 = semis, 8 = final):
```
Bracket bmqv0c1qm - round 1, match 1
bkt|1|bmqv0c1qm|4|1|0|2|0|
```
```
Bracket bmqv0c1qm - round 2, match 1
bkt|1|bmqv0c1qm|4|2|0|2|1|6,7
```

The client extracts the line matching `/^bkt\|1\|/` from the description.
**A tournament with no such line (and no `Bracket …` prose) is not part of a
bracket — render it as a normal single card.**

### Option C (optional hardening): verify against on-chain gating

For trust-minimised reconstruction, the feeder edges can be cross-checked
on-chain: each gated match's `entry_requirement` is a `tournament_validator`
extension whose config span is `[qualifierType, qualifyingMode, topPositions,
…feederTournamentIds]`. The API exposes `entryRequirement`; decoding the config
yields the same feeder ids as `feederTids`. Use this only if you want to
validate the descriptor (not required for display).

## Client implementation

### 1. Grouping
- Fetch tournaments as today (the sepolia API is `budokan-api-sepolia…`, mainnet
  `budokan-api-production…`; both expose `name`, `description`, `entryRequirement`,
  prizes, leaderboard, phase).
- Parse each tournament's descriptor (Option B line, fallback to Option A prose).
- Bucket by `bracketId`. Tournaments with no descriptor → standalone cards
  (unchanged behaviour).

### 2. Tree reconstruction
- Sort a bracket's matches by `(round, matchIndex)`.
- Edges: a match's children are the tournaments whose ids are in its
  `feederTids` (Option B), or matches `2m`/`2m+1` of `round−1` (Option A).
- Root = the match with `isFinal = 1` (or `round = rounds`).

### 3. Collapsed card
Show one card per bracket:
- **Title:** `<gamePrefix> — Bracket` (gamePrefix from any match name, before `R`).
- **Size:** `N players`.
- **Status** (derive — see §5): `Registering` / `Round k of R live` / `Complete`.
- **Prize pool:** aggregate of all prizes across the bracket's matches, per
  token (e.g. `120 STRK`), optionally with the placement split.
- **Champion:** shown when complete (winner of the final).
- A control to expand into the tree.

### 4. Expanded card — the tree
- Render `rounds` columns (Round 1 … Final). Each match = a node showing its two
  competitors, their scores, the winner (highlight), and a link to that match's
  tournament page (`/tournament/<id>?network=<chain>`).
- Highlight **live** matches and the current round.
- An empty/placeholder competitor (under-filled bracket) renders as a **bye /
  walkover** — the present player advances.

### 5. Derived fields
- **Status:** `Complete` if the final has a position-1 finisher; else
  `Round k live` where k is the highest round with a created/active match; else
  `Registering` (round-1 registration still open and not all slots entered).
- **Champion:** the final match's leaderboard position 1 (address → name via the
  usual lookup).
- **Per-match winner:** leaderboard position 1 of that match.
- **Prize pool:** sum prize amounts across all matches, grouped by token; the
  placement breakdown maps to positions (final p1 = 1st, final p2 = 2nd, each
  semi p2 = tied-3rd, each QF p2 = tied-5th, …).
- **Progress:** count rounds whose matches are all resolved.

## Edge cases
- **Under-filled bracket / byes:** a match with fewer than 2 entrants — the
  present competitor walks over; an empty slot shows as a bye. (The bot fills
  slots in join order; unfilled slots stay empty.)
- **In-progress:** all match tournaments are created up front (open + closed
  brackets both deploy the whole tree), so the full structure is visible
  immediately; matches light up round by round.
- **Multiple concurrent brackets:** independent `bracketId`s → independent cards.
- **Mixed prize tokens:** seed and entry fees may be different tokens; aggregate
  the pool **per token** and list each.
- **Non-bracket tournaments:** no descriptor → render exactly as today.
- **Truncation:** `name` is ≤31 chars, so derive `gamePrefix`/round/match from
  the **description/descriptor**, not the name, when in doubt.

## Producer change (budokan-sdk) — needed only for Option B

- **File:** `src/brackets/index.ts`, in `gatedMatchCreateCall` (and the round-1
  create path), where `description` is built.
- Append the `bkt|1|…` line to `description`. The function already has: bracket
  id (`state.id`), size (`state.size`/`bracketSize`), round + `indexInRound`,
  total rounds (`bracketRounds(state)`), and — for round ≥2 — the feeder
  tournament ids (`bracketFeeders(state, match.id)` → their `tournamentId`s,
  which are attached before the later round is created). `isFinal = round ===
  bracketRounds(state)`.
- `description` is a ByteArray (multi-felt), so the extra line is free of the
  31-char `name` limit.
- Bump the SDK minor version; the client keys off the new line and falls back to
  the prose for older brackets.
- (Alternative: pack `bracketId`-hash + round + match into the `metadata_value`
  felt for a numeric handle. Not recommended — the string descriptor is simpler
  and human-debuggable.)

## Acceptance criteria
- A deployed 4 / 8 / 16-player bracket renders as **one** card in the listing.
- Expanding shows the correct tree: round-1 pairings, feeder→match edges, the
  final, and per-match winners/scores.
- The champion appears when the final resolves; the pool aggregates across
  matches (per token) with the placement split.
- Standalone (non-bracket) tournaments are unaffected.
- Works on both sepolia and mainnet (only the API base + `network` param differ).

## Test fixtures (4-player bracket)
Three tournaments with descriptions:
```
Bracket demo123 - round 1, match 1   ·  bkt|1|demo123|4|1|0|2|0|
Bracket demo123 - round 1, match 2   ·  bkt|1|demo123|4|1|1|2|0|
Bracket demo123 - round 2, match 1   ·  bkt|1|demo123|4|2|0|2|1|<tid r1m1>,<tid r1m2>
```
Expected: one card "Bracket (4 players)"; expanded tree = two semis feeding the
final; champion = position 1 of the final once resolved.
